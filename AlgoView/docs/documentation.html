1. Overview

AlgoView is a pure front-end web application with no backend, no build step, and no external dependencies. It runs entirely in the browser by opening index.html with a local server (e.g., VS Code Live Server).

No installation required. No npm install. No backend server. Just open index.html in a browser.

How to Run

  • Install Live Server extension in VS Code (by Ritwick Dey).
  • Right-click index.html → Open with Live Server.
  • Or simply double-click index.html to open directly in your browser.

4. Algorithm Catalog

4.1 Sorting (6 algorithms)

🫧 Bubble Sort

Repeatedly compares and swaps adjacent elements until sorted.

Best: O(n)  |  Avg: O(n²)  |  Space: O(1)
stablein-place

🎯 Selection Sort

Finds the minimum in the unsorted part and places it at the front.

Best/Avg/Worst: O(n²)  |  Space: O(1)
unstablein-place

🃏 Insertion Sort

Builds a sorted array one element at a time by inserting into position.

Best: O(n)  |  Avg: O(n²)  |  Space: O(1)
stableadaptive

🔀 Merge Sort

Divide-and-conquer: splits, sorts recursively, then merges.

Best/Avg/Worst: O(n log n)  |  Space: O(n)
stablerecursive

⚡ Quick Sort

Picks a pivot, partitions the array, then recurses on sub-arrays.

Avg: O(n log n)  |  Worst: O(n²)  |  Space: O(log n)
unstablein-place

🌳 Heap Sort

Builds a max-heap, then repeatedly extracts the maximum.

Best/Avg/Worst: O(n log n)  |  Space: O(1)
unstablein-place

4.2 Searching (2 algorithms)

🔦 Linear Search

Scans each element sequentially until the target is found or exhausted.

Best: O(1)  |  Avg/Worst: O(n)  |  Space: O(1)
sequentialsimple

🎯 Binary Search

Halves the search space each step. Requires a sorted array.

Best: O(1)  |  Avg/Worst: O(log n)  |  Space: O(1)
sortedfast

4.3 Data Structures (4)

📚 Stack

LIFO structure. Demonstrates push, pop, and peek operations.

All operations: O(1)  |  Space: O(n)
LIFO

🚌 Queue

FIFO structure. Demonstrates enqueue and dequeue operations.

All operations: O(1)  |  Space: O(n)
FIFO

🔗 Linked List

Nodes linked by pointers. Demonstrates append, insert, and delete.

Access: O(n)  |  Insert at node: O(1)
dynamic

🌲 Binary Search Tree

Hierarchical tree with insert and search operations visualized.

Avg: O(log n)  |  Worst: O(n)
BSTtree

4.4 Graph Algorithms (3)

🌊 Breadth-First Search

Explores level-by-level using a queue. Finds shortest path in unweighted graphs.

O(V+E) time  |  O(V) space
traversalshortest-path

🕳 Depth-First Search

Explores as deep as possible before backtracking. Uses a stack/recursion.

O(V+E) time  |  O(V) space
traversalrecursive

🗺 Dijkstra's Algorithm

Finds shortest paths from source in a weighted graph using a greedy approach.

O((V+E) log V)  |  O(V) space
weightedgreedy

🧩 Fibonacci (Dynamic Programming)

Computes Fibonacci numbers using memoization (DP table), showing how overlapping subproblems are cached to avoid recomputation.

O(n) time  |  O(n) space
DPmemoization

5.2 Custom Array Input

  • Type a comma-separated list of numbers (e.g. 64, 34, 25, 12, 22) and click Apply.
  • Maximum of 20 values are accepted.
  • Click 🎲 Random to generate a random array of 6–13 elements.
  • For searching algorithms, the input field accepts the target value.

5.3 Code Panel

  • Displays pseudocode for the currently selected algorithm.
  • The active line is highlighted in real-time as each frame plays.
  • Automatically scrolls to keep the highlighted line in view.

5.4 Info Panel

  • Step Log — Keeps a running log of the last 30 step messages explaining what is happening.
  • Complexity Table — Shows Best / Average / Worst time and space complexity for the selected algorithm.
  • About — A plain-English description of how the algorithm works.

5.5 Canvas Overlays

  • Comparison counter — Tracks total comparisons made so far.
  • Swap counter — Tracks total swaps made (sorting only).
  • Step indicator — Shows current step out of total (e.g. 5 / 42).
  • Progress bar — Visual progress across the full animation.
  • Color legend — Explains bar/node colors (Default, Comparing, Swapping, Sorted).

7. Technical Architecture

7.1 Generator-Based Frame System

Every algorithm is implemented as a JavaScript generator function. Each yield statement emits a frame — a snapshot of the data structure's state at that moment. The visualizer pre-collects all frames into an array, enabling:

  • Instant step-forward and step-backward navigation
  • Smooth playback at any speed without re-running the algorithm
  • Accurate step counter and progress bar
// Example frame structure (sorting)
{
  array:       [64, 25, 34, 12],   // current array values
  states:      ['sorted','comparing','default','comparing'],
  comparisons: 5,
  swaps:       2,
  codeLine:    3,                  // which pseudocode line to highlight
  message:     "Comparing 25 with 34",
  type:        'sorting'
}

7.2 Renderer System

Two canvas renderers handle all visual output:

  • ArrayRenderer — Draws bar charts for sorting/searching, horizontal nodes for Stack/Queue/Linked List, and tree layouts for BST.
  • GraphRenderer — Draws nodes and edges for BFS, DFS, and Dijkstra on a fixed-position 8-node graph. Also renders the Fibonacci DP table.

Renderers are called on every frame via ResizeObserver so the canvas always fills its container correctly.

7.3 Algorithm Registry

All algorithm metadata is stored in registry.js as the global ALGO_REGISTRY array. Each entry contains: id, name, category, emoji, description, complexity, tags, and type. Both the home page and visualizer sidebar read from this single source of truth.

7.4 URL State

When an algorithm is selected, the URL updates to ?algo=bubble-sort (etc.) via history.replaceState. This means you can share or bookmark a direct link to any algorithm.
